🔄 Runbook: Redémarrage backend détecté

Alerte: Redémarrage détecté sur open-hub-backend
Type: Information - Monitoring

📋 Description

Cette alerte se déclenche quand l'application backend a redémarré. C'est une alerte informative qui peut indiquer :

  • Un redémarrage manuel volontaire
  • Un crash suivi d'un redémarrage automatique
  • Un déploiement (normal)
  • Un problème de stabilité

🔍 Investigation

1. Vérifier la cause du redémarrage

# Logs PHP-FPM
grep "reload\|restart\|exit" /var/log/php-fpm.log -B5 -A5

# Logs système (si accès)
sudo journalctl -u php-fpm --since "5 minutes ago"

# Logs Render
https://dashboard.render.com/web/open-hub-backend/logs

2. Vérifier si c'était un crash

# Chercher des erreurs fatales avant le redémarrage
grep -i "fatal\|error\|exception\|segfault" /var/log/php.log -B10
Indice Conclusion Action
Logs propres, redémarrage planifié Redémarrage normal (déploiement) RAS - Surveiller
Erreurs fatales avant redémarrage Crash application Analyser l'erreur
"Killed" dans logs Manque de mémoire Vérifier ressources
"Maximum execution time" Script trop long Optimiser requête

📊 Vérifications post-redémarrage

Santé de l'application

# Vérifier health endpoint
curl -I https://open-hub-backend.onrender.com/health
curl https://open-hub-backend.onrender.com/api/v1/health

Métriques

# Grafana - Vérifier après redémarrage
- Taux d'erreurs 5xx
- Temps de réponse
- Utilisation mémoire
- Nombre de requêtes

Base de données

# Vérifier que les connexions DB sont rétablies
# Les logs ne doivent pas montrer d'erreurs de connexion

📝 Documentation

Si le redémarrage était anormal :

  • Noter l'heure exacte du redémarrage
  • Documenter l'erreur qui a causé le crash
  • Créer un ticket pour corriger le problème
  • Surveiller si le problème se reproduit
