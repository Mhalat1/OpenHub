💀 Runbook: Backend hors ligne (Down)

CRITIQUE - P1 - URGENT
🚨 ALERTE: PHP est down - Instance backend hors ligne
Impact: Application totalement inaccessible
Instance: openhub-backend.onrender.com
⚠️ IMPORTANT: Cette alerte signifie que l'application ne répond plus du tout. Action immédiate requise !

⚡ 1. Actions immédiates (T0 - 2 minutes)

🔍 Vérifier si l'instance répond

# Tester l'endpoint health
curl -I https://openhub-backend.onrender.com/health
curl -I https://openhub-backend.onrender.com/

# Vérifier le code HTTP (doit être 200)
# Si 502/503/504 → Application crashée
# Si 000 → DNS/Network

📊 Vérifier Render Dashboard

1. Aller sur https://dashboard.render.com
2. Naviguer vers "openhub-backend"
3. Vérifier le statut du service:
   - 🔴 Down → Redémarrer immédiatement
   - 🟡 Degraded → Vérifier logs
   - 🔵 Deploying → Attendre fin déploiement

📝 Vérifier les logs récents

# Via Render Dashboard
Logs → Voir les dernières lignes
Chercher: "Fatal error", "Exception", "Killed", "Out of memory"

# Causes fréquentes dans logs:
- "Allowed memory size exhausted" → Mémoire insuffisante
- "Maximum execution time exceeded" → Script trop long
- "Class not found" → Autoloader cassé
- "Connection refused" → Base de données inaccessible

🔄 2. Redémarrage d'urgence

Via Render Dashboard:

1. Services → openhub-backend
2. Cliquer sur les "..." (3 points)
3. Sélectionner "Restart"
4. Confirmer le redémarrage
5. Attendre 30 secondes
6. Rafraîchir et vérifier le statut

Via Render CLI (si configuré):

render restart --service openhub-backend
render ps --service openhub-backend  # Vérifier statut

Vérification post-redémarrage:

# Vérifier que l'API répond
curl https://openhub-backend.onrender.com/health
curl https://openhub-backend.onrender.com/api/v1/health

# Vérifier les logs pour confirmer démarrage
# Doit voir: "Server started", "Listening on port"

🔍 3. Diagnostic des causes

Cause possible Indices Solution
💥 Crash PHP Logs: "segfault", "core dumped" Redémarrer + vérifier code récent
🧠 Mémoire insuffisante Logs: "Out of memory", "Killed" Augmenter memory_limit ou scale up
🗄️ Base de données HS Logs: "Connection refused", "SQLSTATE" Vérifier statut DB
📦 Déploiement cassé Erreurs juste après déploiement Rollback à version précédente
🌐 Render en panne Status.render.com indique incident Attendre, contacter support
⚙️ Configuration invalide Fichier .env manquant Vérifier variables d'environnement

🛠️ 4. Procédures avancées

📦 Rollback de déploiement

# Si le problème vient d'un déploiement récent
1. Render Dashboard → Deployments
2. Identifier le dernier déploiement stable
3. Cliquer sur "Rollback" à côté de cette version
4. Confirmer le rollback
5. Vérifier que l'application redémarre correctement

# Alternative via Git
git log --oneline -10
git checkout [stable-commit-hash]
git push -f origin main  # Force push si nécessaire

💾 Vérification base de données

# Tester connexion DB depuis Render
# Utiliser Render Shell ou une tâche one-off

# Via psql (PostgreSQL)
psql $DATABASE_URL -c "SELECT 1"

# Via MySQL
mysql $DATABASE_URL -e "SELECT 1"

# Vérifier que les migrations sont appliquées
php artisan migrate:status  # Laravel
php bin/console doctrine:migrations:status  # Symfony

🔧 Debug PHP avancé

# Activer l'affichage des erreurs temporairement
# Dans index.php ou le point d'entrée
ini_set('display_errors', 1);
error_reporting(E_ALL);

# Vérifier la configuration PHP
php -m  # Modules chargés
php -i | grep memory_limit

# Tester le chargement des classes
php -r "require 'vendor/autoload.php'; echo 'Autoload OK';"
